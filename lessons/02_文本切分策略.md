# 02 · 文本切分策略

> 对应代码：`1.按照固定字符数切分.py` / `2.按固定字符数 结合overlapping window.py` / `3.递归方法 RecursiveCharacterTextSplitter.py`

---

## 为什么要切分文本？

向量数据库检索时，每次只能处理有限长度的文本片段。  
一篇 10 万字的文档不能整体向量化——它太长了，向量会"平均化"所有信息，导致什么都检索不到。

**目标：** 把长文档切成语义尽量完整的小片段（chunk），每个 chunk 独立向量化和检索。

---

## 策略一：固定字符数切分

最简单的做法：每 N 个字符截一刀。

```python
def split_by_fixed_char_count(text, count):
    return [text[i:i + count] for i in range(0, len(text), count)]

chunks = split_by_fixed_char_count(text, 100)
```

**可视化：**

```
原文：  [----100字----][----100字----][----100字----][--剩余--]
切分后：  chunk0          chunk1          chunk2          chunk3
```

**问题：** 完全不考虑语义，一句话可能被切断：

```
chunk0: "...镜头的焦距计算方法是根据工作距离和传感器尺寸来确
chunk1: 定的，具体公式为 f = WD × sensor_size / FOV..."
```

用户问"焦距计算公式"时，chunk0 和 chunk1 各含半句话，单独检索都是残缺的。

---

## 策略二：滑动窗口（Overlap）

在固定切分的基础上，让相邻 chunk 之间**重叠**一段内容，避免关键信息落在边界上。

```python
def sliding_window_chunks(text, count, stride):
    # stride = count - overlap
    return [text[i:i + count] for i in range(0, len(text), count - stride)]

chunks = sliding_window_chunks(text, 100, stride=20)  # 重叠 20 字符
```

**可视化：**

```
原文位置:  0        100      200
chunk0:   [====100字====]
chunk1:          [====100字====]   ← 与 chunk0 重叠 20 字
chunk2:                  [====100字====]
```

**改善：** 边界处的完整句子现在至少有一个 chunk 能完整包含它。  
**代价：** 存储量增加，约为 `1 / (1 - overlap_ratio)` 倍。

---

## 策略三：递归字符切分（生产推荐）

LangChain 内置的 `RecursiveCharacterTextSplitter` 是目前最常用的通用切分方案。

**核心思路：** 按优先级依次尝试不同的分隔符，优先在语义边界处切割：

```
优先级：段落（\n\n）→ 换行（\n）→ 句号（。）→ 空格（ ）→ 强制截断
```

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,
    chunk_overlap=50,
    separators=["\n\n", "\n", "。", "！", "？", " ", ""]
)
chunks = splitter.create_documents([text])
```

**执行过程示例：**

```
文本：
  段落A（300字）\n\n段落B（600字）\n\n段落C（200字）

第一步：按 \n\n 切分 → [段落A, 段落B, 段落C]
第二步：段落B 超过 500字，再按 \n 切分 → [段落B前半, 段落B后半]
第三步：所有 chunk ≤ 500字，完成
```

---

## 三种策略对比

| 策略 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 固定字符 | 实现简单 | 随意断句，语义破碎 | 调试、理解概念 |
| 滑动窗口 | 减少边界截断 | 冗余存储，重复检索 | 短文档、对召回要求高 |
| 递归切分 | 尊重自然语义边界 | chunk 大小不均匀 | **绝大多数场景** |

---

## 参数选择指南

**chunk_size：**
- 太小（< 200）：单个 chunk 信息量不足，LLM 无法作答
- 太大（> 2000）：向量语义被稀释，检索精度下降
- 推荐：**500 ~ 1200 字**（根据文档类型调整）

**chunk_overlap：**
- 通常设为 chunk_size 的 10% ~ 20%
- 主项目使用 overlap=0，配合 Markdown 标题切分保证语义完整性

## 主项目中的对应代码

`splitters.py` 实现了 5 种策略（`markdown`、`regex`、`semantic`、`recursive`、`auto`），  
并通过 `kb_config.json` 让每个知识库目录可以独立配置策略。
