## 多轮对话的挑战

大模型本身是**无状态的**——每次调用都是独立的，它不记得上一轮说了什么。  
要实现"你刚才说的公式能再解释一下"这样的对话能力，  
必须把历史消息手动拼接进每次请求的 Prompt 中。

```
第 1 轮请求：[system] + [user: "镜头焦距怎么算"]
第 2 轮请求：[system] + [user: "镜头焦距怎么算"] + [ai: "..."] + [user: "你刚才的公式能解释吗"]
第 3 轮请求：[system] + [user: ...] + [ai: ...] + [user: ...] + [ai: ...] + [user: "新问题"]
```

对话越长，每次发送的 Token 越多，成本越高，且有上下文窗口限制。

---

## Token 预算截取

本项目允许的最大历史 Token 数为 3000（`MAX_HISTORY_TOKENS`）。  
截取策略是**从最新一轮往前保留**，丢弃较早的历史（保留最近的上下文更有价值）：

```python
def parse_history(history_data):
    """
    前端传来的 [{role, content}] → LangChain 消息对象列表
    从最新一条往前计数，总 Token 数不超过 MAX_HISTORY_TOKENS
    """
    items = list(reversed(history_data or []))    # 倒序，从最新开始
    total_tokens = 0
    selected = []

    for item in items:
        role    = item.get('role', '')
        content = item.get('content', '')
        t = _count_tokens(content)

        if total_tokens + t > Config.MAX_HISTORY_TOKENS:
            break                    # 超过预算，停止添加

        selected.insert(0, (role, content))   # 插入到头部，恢复正序
        total_tokens += t

    # 转换为 LangChain 消息对象
    msgs = []
    for role, content in selected:
        if role == 'user':
            msgs.append(HumanMessage(content=content))
        elif role == 'assistant':
            msgs.append(AIMessage(content=content))
    return msgs
```

## Token 计数

精确计数使用 `tiktoken`（OpenAI 官方分词器），不可用时降级为字符数估算：

```python
try:
    import tiktoken
    _enc = tiktoken.get_encoding("cl100k_base")    # GPT-4 / DeepSeek 都用这个编码

    def _count_tokens(text: str) -> int:
        return len(_enc.encode(text))

except ImportError:
    def _count_tokens(text: str) -> int:
        return len(text) // 2    # 中文约 2 字/token（粗略估算）
```

**为什么用 `cl100k_base`？**  
DeepSeek 和许多开源模型的分词器与 GPT-4 的 `cl100k_base` 高度兼容，  
用它估算 Token 数误差很小（< 5%）。

## LangChain 消息对象

```python
from langchain_core.messages import HumanMessage, AIMessage

# HumanMessage = 用户消息
# AIMessage = 模型回复
# SystemMessage = 系统指令（本项目由 Prompt 模板管理）

history = [
    HumanMessage(content="镜头焦距怎么算？"),
    AIMessage(content="焦距 f = WD × sensor_size / FOV，其中..."),
    HumanMessage(content="WD 是什么意思？"),
]
```

这些对象传入 `MessagesPlaceholder("history")` 后，LangChain 会自动展开成对应的消息格式，  
DeepSeek / 千问 / OpenAI 都能正确理解。

## 前端的职责

本项目的对话历史**由前端（浏览器）维护**，每次请求时从 `localStorage` 读取并传给后端：

```json
POST /api/chat/stream
{
    "query": "你刚才的公式再解释一下",
    "history": [
        {"role": "user",      "content": "镜头焦距怎么算？"},
        {"role": "assistant", "content": "焦距 f = WD × sensor_size / FOV ..."},
        {"role": "user",      "content": "WD 是什么意思？"},
        {"role": "assistant", "content": "WD 是工作距离（Working Distance）..."}
    ]
}
```

**这种设计的优缺点：**

| 优点 | 缺点 |
|------|------|
| 服务器无状态，易于横向扩展 | 浏览器清除缓存后历史丢失 |
| 不需要数据库存储历史 | 每次请求携带历史，带宽略增 |
| 用户隐私：数据不上传服务器 | 多设备不同步 |

---

## Token 预算的重要性

假设不截取历史，对话进行到第 20 轮时，历史可能已经 8000 Token，  
加上参考资料（约 1500 Token）+ 问题（约 50 Token），  
总输入已超过许多模型的上下文窗口（4096 Token），请求会**报错**。

**预算截取保证了系统在任何长度的对话下都能稳定运行。**
