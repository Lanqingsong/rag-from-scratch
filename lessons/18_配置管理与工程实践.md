## 配置的三层级结构

本项目的配置采用"优先级覆盖"模式，越靠后优先级越高：

```
代码默认值（config.py 类属性）
    ↑ 被覆盖
.env 文件（dotenv 在启动时加载）
    ↑ 被覆盖
model_config.json（用户在 Web 界面保存的运行时配置）
```

```python
class Config:
    # 层级 1：代码默认值
    MODEL_NAME  = "deepseek-v4-pro"
    TEMPERATURE = 0.7

    @classmethod
    def load_runtime(cls):
        # 层级 3：读取 model_config.json 覆盖
        if not os.path.exists(_RUNTIME_CONFIG_PATH):
            return
        with open(_RUNTIME_CONFIG_PATH) as f:
            d = json.load(f)
        if d.get("model_name"):
            cls.MODEL_NAME = d["model_name"]     # 覆盖了类属性

# 模块加载时立即读取（层级 2 由 load_dotenv() 在文件顶部完成）
Config.load_runtime()
```

**设计原理：**
- `.env` 存放密钥等机密信息（不提交 git）
- `model_config.json` 存放运行时调整的参数（不提交 git）
- 代码默认值作为最后的保底，确保不配置也能运行

## API Key 安全处理

前端展示 API Key 时需要脱敏，同时保证"用户未修改时不破坏原有 Key"：

```python
@classmethod
def get_runtime(cls) -> dict:
    """返回当前配置，API Key 脱敏"""
    key = raw.get("api_key") or cls.DEEPSEEK_API_KEY or ""
    # 脱敏：sk-****************xxxx
    masked = ("sk-" + "*" * 16 + key[-4:]) if len(key) > 8 else ("*" * len(key))
    return {
        "api_key":    masked,      # 前端显示脱敏后的值
        "has_key":    bool(key),   # 前端用于判断是否已配置
        "model_name": ...,
    }
```

**保存时的反脱敏：**

```python
# app.py 保存配置时
if data['api_key'].startswith('sk-') and '*' in data['api_key']:
    # 用户没改 Key（提交的还是脱敏版），保留数据库中的原始值
    data['api_key'] = existing_config.get('api_key', Config.DEEPSEEK_API_KEY)
```

## 可选依赖的优雅降级

本项目有两个非必须依赖，缺失时系统正常运行，只是功能受限：

```python
# dashscope（千问 Embedding + Reranker）
try:
    import dashscope
    HAS_DASHSCOPE = True
except ImportError:
    HAS_DASHSCOPE = False
    # 自动降级到 OpenAI Embeddings 接口（DeepSeek 兼容）

# rank_bm25（BM25 关键词检索）
try:
    from langchain_community.retrievers import BM25Retriever
    HAS_BM25 = True
except ImportError:
    HAS_BM25 = False
    print("Warning: rank_bm25 未安装，BM25 混合检索已禁用")
    # 系统继续运行，只用 FAISS 单路检索
```

**这种模式的好处：**
- 降低上手门槛（没有千问 Key 也能跑起来）
- 生产环境按需安装，减少镜像体积

## 启动校验

服务启动时立即验证必要配置，而不是等到第一次请求时才报错：

```python
@classmethod
def validate(cls):
    if not cls.DEEPSEEK_API_KEY:
        raise ValueError("DEEPSEEK_API_KEY 未设置，请检查 .env 文件")
    try:
        import dashscope
        if not cls.QIANWEN_API_KEY:
            raise ValueError("QIANWEN_API_KEY 未设置")
    except ImportError:
        pass    # 没有安装 dashscope，跳过千问 Key 校验
    return True

# app.py 启动时
try:
    Config.validate()
except ValueError as e:
    print(f"配置错误: {e}")
    sys.exit(1)    # 配置错误直接退出，不启动服务
```

## Windows 编码问题处理

Python 在 Windows 上默认使用 GBK 编码，而中文日志可能包含 UTF-8 字符，  
导致 `UnicodeEncodeError`：

```python
# app.py 和 main.py 开头
if sys.platform == 'win32':
    sys.stdout.reconfigure(encoding='utf-8', errors='replace')
    sys.stderr.reconfigure(encoding='utf-8', errors='replace')
```

`errors='replace'` 意味着无法编码的字符用 `?` 替代，而不是崩溃。

## 项目工程结构的设计原则

回顾整个项目，每个文件的职责划分：

| 文件 | 职责 | 设计原则 |
|------|------|---------|
| `config.py` | 配置中枢 | 所有参数集中一处，其他模块只 import Config |
| `knowledge_base.py` | 数据层 | 文档加载、向量化、检索，不处理 HTTP |
| `splitters.py` | 分块层 | 只负责切分，不依赖其他业务模块 |
| `llm_client.py` | LLM 层 | 只管 LLM 调用，不管检索 |
| `prompts.py` | 提示词层 | 模板定义和热重载，与其他逻辑解耦 |
| `web_search.py` | 网络层 | 只管搜索，含自己的容错（熔断器） |
| `app.py` | 接入层 | 路由、协调各模块、SSE 推送 |
| `main.py` | CLI 入口 | 命令行交互，复用核心模块 |

**关键原则：单一职责**——每个文件只做一件事，修改一个功能时只需改一个文件。

## .env 文件的最佳实践

```bash
# 项目根目录
.env          ← 真实 Key，列入 .gitignore，绝不提交
.env.example  ← 占位符，提交到 git，告诉别人需要哪些变量
```

```env
# .env.example（提交到 git）
DEEPSEEK_API_KEY=sk-xxxxxxxxxxxxxxxxxxxx
QIANWEN_API_KEY=sk-xxxxxxxxxxxxxxxxxxxx
SEARXNG_URL=http://localhost:8080

# .env（本地，不提交）
DEEPSEEK_API_KEY=sk-abc123实际密钥def456
```
