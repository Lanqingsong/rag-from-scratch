# 16 · 网络搜索与熔断器模式

> 对应主项目 `web_search.py` 全文

---

## 为什么需要网络搜索？

本地知识库的局限性：

- **知识截止日期：** 文档上次更新后的新进展不知道
- **覆盖不完整：** 用户问了一个知识库没有收录的问题

本项目引入 SearXNG 自托管搜索引擎作为补充，  
当本地知识库命中质量不高时，自动联网检索最新内容。

---

## SearXNG

[SearXNG](https://docs.searxng.org/) 是开源的**元搜索引擎**：  
它把你的搜索请求同时发给 Bing、Google、DuckDuckGo 等多个引擎，  
聚合结果后返回给你，**全程不追踪用户数据**。

本地自托管（Docker）的好处：
- 免费，无 API 限额
- 数据不经过第三方服务器
- 可选择启用/禁用哪些搜索引擎

```bash
# 一键启动
docker run -d --name searxng -p 8080:8080 searxng/searxng
```

## SearXNG 集成代码

```python
from langchain_community.utilities import SearxSearchWrapper

wrapper = SearxSearchWrapper(
    searx_host="http://localhost:8080",
    k=3,    # 每次返回 3 条结果
)

def _do_search(self, query):
    raw = self._wrapper.results(
        query,
        num_results=3,
        language="zh-CN",
    )
    docs = []
    for r in raw:
        snippet = r.get("snippet") or r.get("content", "")
        title   = r.get("title", "")
        link    = r.get("link") or r.get("url", "")
        docs.append(Document(
            page_content=f"{title}\n{snippet}".strip(),
            metadata={"source": link, "title": title, "type": "web"}
        ))
    return docs
```

## 熔断器模式（Circuit Breaker）

**问题：** 如果 SearXNG 服务挂了，每次搜索都会报错、等待超时，严重影响主流程。

**熔断器的思路：** 像保险丝一样——连续失败超过阈值就"断开"，  
停止尝试一段时间（恢复期），等服务恢复后再"闭合"。

```
正常状态（Closed）→ 记录每次失败 → 失败 N 次 → 断开状态（Open）
                                                         ↓
                                              暂停 timeout 秒
                                                         ↓
                                              尝试恢复（Half-Open）
                                                         ↓
                                     成功 → 闭合；失败 → 重新断开
```

---

## 熔断器实现

```python
class _CircuitBreaker:
    def __init__(self, failure_threshold=3, recovery_timeout=60):
        self._failures   = 0
        self._threshold  = failure_threshold   # 连续失败 3 次触发熔断
        self._timeout    = recovery_timeout    # 断开后 60 秒自动恢复
        self._opened_at  = None
        self._lock       = Lock()              # 线程安全

    @property
    def is_open(self):
        with self._lock:
            if self._opened_at is None:
                return False                   # 未熔断
            if time.time() - self._opened_at > self._timeout:
                self._opened_at = None
                self._failures  = 0
                return False                   # 超过恢复时间，自动闭合
            return True                        # 仍在熔断期

    def record_failure(self):
        with self._lock:
            self._failures += 1
            if self._failures >= self._threshold:
                self._opened_at = time.time()  # 触发熔断，记录断开时间
                print(f"[WebSearch] 熔断器打开，暂停 {self._timeout}s")

    def record_success(self):
        with self._lock:
            self._failures  = 0
            self._opened_at = None             # 成功后重置
```

## 指数退避重试

在触发熔断之前，先重试 3 次，等待时间按指数增长：

```python
def search(self, query):
    if self._breaker.is_open:
        return []          # 熔断中，直接跳过

    last_exc = None
    for attempt in range(3):
        try:
            results = self._do_search(query)
            self._breaker.record_success()
            return results
        except Exception as e:
            last_exc = e
            wait = 2 ** attempt    # 1s, 2s, 4s
            print(f"[WebSearch] 第 {attempt+1} 次失败，{wait}s 后重试")
            if attempt < 2:
                time.sleep(wait)

    self._breaker.record_failure()   # 3 次都失败，计入熔断计数
    return []
```

**指数退避的意义：**  
立刻重试往往还是失败（服务仍在崩溃恢复中）；  
等待时间加倍，给服务足够的时间恢复，同时避免重试风暴（多个客户端同时轰炸）。

## 并行检索设计

KB 检索和网络搜索并行执行，互不等待：

```python
# app.py
def _parallel_search(query):
    if web_searcher:
        with ThreadPoolExecutor(max_workers=2) as ex:
            kb_f  = ex.submit(kb.search, query, Config.TOP_K_RESULTS, llm.llm)
            web_f = ex.submit(web_searcher.search, query)
            return kb_f.result(timeout=20), web_f.result(timeout=15)
    return kb.search(query, llm=llm.llm), []
```

- KB 检索超时 20 秒（含 Reranker API 时间）
- 网络搜索超时 15 秒
- 任一超时会抛出异常，但不影响另一路结果
